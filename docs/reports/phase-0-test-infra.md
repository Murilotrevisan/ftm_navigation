## What was built

The container and the five test harnesses, each one executed until it passed.

- **`docker/`** — `Dockerfile` (pinned `espressif/idf:v5.5.2` + Ruby/Ceedling +
  gcovr), `docker-compose.yml` (repo bind mount, persistent ccache volume, UID
  variables, **no device mapping, not privileged**), `entrypoint.sh`, `README.md`.
- **`tools/`** — `dev.ps1` (Windows entry point, routes each subcommand to
  container or venv), `dev.sh` (container side), `boards.json` (role → MAC),

  E2E fixtures).

  (core): the real modules L1a tests, including a mockable dependency.
- **`components/services/`** — `ftm_quality`: the module L1b tests, encoding
  the clamp signature from `HARDWARE_FINDINGS.md` §4/§6.
- **`tests/`** — `host_ceedling/` (L1a), `host_idf/` (L1b), `target_smoke/`
  (L2 template + the known-good app L3 flashes), `e2e/` (L3), `tools/` (L4),
  `sim/` and `manual/` (placeholders with READMEs), `pytest.ini`, `README.md`.

## Deviations from the phase document

Five, all deliberate.

1. **Coverage reports are generated by an explicit `gcovr` call, not by
   Ceedling's gcov plugin.** The plugin runs gcovr with `report_root`
   defaulting to the working directory, which cannot see coverage data written
   to an out-of-tree `build_root`. It produced a report reading
   `lines-valid="0"` — the exact failure `CONTAINER.md` §7 warns about, and
   one that looks like a pass. `:reports:` is now empty and `dev.sh coverage`
   invokes gcovr with a root that resolves both the test and the domain
   sources, **and fails if the measured line count is zero**. Evidence in
   criterion 4 below.

2. **UID mapping is present but is not what protects Windows.** The compose
   file exposes `FTM_UID`/`FTM_GID`, defaulting to `0`. On Docker Desktop for
   Windows the bind mount is drvfs: the container runs as root
   (`uid=0(root)`), yet every file it creates surfaces on the host owned by
   `MURILOTREVISAN\murilo`. Running as an arbitrary UID inside
   `espressif/idf` risks an unwritable `HOME` and unreadable tool paths, for
   no benefit on this platform. The mapping is kept for the Linux/WSL path.
   Both halves of that claim are verified in criterion 9.

3. **`tests/host_ceedling/` also contains `test_ftm_result.c` and
   `test_ftm_result_report.c`**, beyond the specified
   `test_harness_smoke.c` — the required worst-case set (out-of-range enum,
   NULL, buffer-too-small with canary, CMock succeed *and* fail) does not fit
   in one smoke file, and splitting them keeps the CMock example copyable.
   Same for `tests/host_idf/main/test_ftm_quality.c`.

4. **`tests/pytest.ini` sets `--import-mode=importlib`.** The deliverables
   name a `test_harness_smoke.py` in both `tests/e2e/` and `tests/tools/`;
   under the legacy import mode two test files sharing a basename collide
   during collection. importlib mode keeps both filenames as specified rather
   than renaming a deliverable or scattering `__init__.py` files.

5. **An L1b harness self-check was added** (`dev.ps1 test-host-selfcheck`,
   compiled in only under `-DFTM_TEST_SELFCHECK_FAIL`). A suite that always
   exits 0 is indistinguishable from a suite that passes, and the linux-target
   hang trap makes that failure mode specifically likely. This is a small
   scope addition; it is what turns "the harness works" from a claim into a
   check.

## Blockers encountered

**One, diagnosed and cleared: COM4 held by another session.** The first `e2e`
run failed with `PermissionError(13, 'Acesso negado.')` on COM4. Before
concluding anything about the harness:

```
> Get-CimInstance Win32_Process -Filter "Name like '%python%'" | Select ProcessId, CommandLine
ProcessId    : 28380
CommandLine  : "...\ftm_measurement\.venv\Scripts\python.exe" tools\bench\validate_board.py
               --sessions 8 --log-dir C:\tmp\ftm-phase1-restored-pass
ProcessId    : 20536
CommandLine  : "...\idf-python\3.11.2\python.exe" tools\bench\validate_board.py --sessions 8 ...
```

Root cause: **Phase 1 was driving the same two boards concurrently.** Not a
fault in this harness. Once that run finished, `e2e` passed unchanged. See
*New findings* — this is a standing hazard, not a one-off.

Two consecutive `e2e` runs immediately afterwards then failed to read COM3's
MAC while the port was otherwise fine (a direct `esptool read_mac` on COM3
succeeded seconds later). I could not reproduce it after adding
instrumentation, so the cause is **not established**; the most likely
explanation is the board still settling from the other session's reset. What
came out of it is permanent: `read_mac` now prints *why* a probe failed
instead of returning a silent `None`, which had turned "attached but busy"
into the misleading message "board not attached".

## New findings

**1. `CONFIG_ESP_WIFI_FTM_ENABLE` is `default n` — `HARDWARE_FINDINGS.md` §1
updated.** The L2 test failed on real hardware with
`CONFIG_ESP_WIFI_FTM_INITIATOR_SUPPORT is disabled`. Confirmed against the IDF
source: the initiator and responder options are `default y` but both
`depends on ESP_WIFI_FTM_ENABLE`, which is `default n`
(`components/esp_wifi/Kconfig:397`). Every earlier measurement used the
Espressif example, whose own `sdkconfig.defaults` sets it, so the dependency
never surfaced. The document previously implied no configuration was needed;
it now states the trap, the generated-`sdkconfig` symptom, and the three lines
every firmware in this repo must carry. **This is the one normative change in
this branch.**

**2. The two boards are a shared, single-user resource.** Phases 0 and 1 are
documented as parallelisable, and in worktrees they are — but they contend for
the same COM ports, and the symptom (`PermissionError`) does not name the
other party. Recorded in `tests/README.md` and `tests/e2e/README.md`.
**Recommendation for the reviewer: hardware-touching phases should not run
concurrently**, or should acquire a lock.

**3. Ceedling 1.1.0 has a fifth config trap** beyond the four in
`CONTAINER.md` §7: `:defines:` must be a hash, and a 0.x-style empty array
fails with `:defines must contain key / value pairs, not array`. Recorded in
`tests/host_ceedling/README.md`.

**4. Ceedling's gcovr plugin cannot see an out-of-tree build root**, and fails
silently into a 0 % report. Details in deviation 1.

**5. `git describe` warns inside the container for a worktree checkout**
(`fatal: not a git repository: /project/C:/Users/...`), because
`.git` in a worktree is a file holding a Windows absolute path. Harmless —
`PROJECT_VER` falls back to `1` — but it will appear in every container build
run from a worktree and is not a broken build.

## Open items

1. **Coverage thresholds are not enforced**, per the decisions in force. There
   is now something real to measure (19 lines); the gate becomes meaningful
   once Phase 3 lands actual modules.
2. **`tests/sim/` and `tests/manual/` are empty.** Their READMEs state the
   required contents. `pytest -m manual` therefore exits 5.
3. **The intermittent MAC-probe failure is instrumented, not solved.** If it
   recurs, `board_ports.read_mac` now prints esptool's reason; a retry with
   backoff is the obvious fix but was not added without evidence that it is
   the right one.
4. **Board contention between phases has no mechanism**, only documentation.
5. **The `ftm_result` / `ftm_quality` modules are Phase 0 reference modules.**
   Phase 3 owns the final designs. The tests encode behaviour that should
   survive redesign, and both headers say so — but a Phase 3 agent should read
   them as "this is the standard to meet", not "this is the finished API".
6. **`pytest-embedded` emits `PytestExperimentalApiWarning:
   record_xml_attribute` once per test.** Cosmetic noise from the plugin, not
   from this code; left unsuppressed rather than hidden behind a filter.
